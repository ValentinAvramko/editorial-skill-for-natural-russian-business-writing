# Контракт адаптера

Этот файл задаёт минимальный контракт для любого адаптера доставки в репозитории.

Сейчас в репозитории есть два активных адаптера:

- `skill` для agent skills;
- `prompt` для ручного использования в обычных чатах.

## Цель

Адаптер должен быть тонким слоем упаковки вокруг одного и того же редакторского стандарта.

Адаптер нужен не для новой prompt-ветки, а для доставки одного и того же стандарта из `core/` в другом runtime-формате.

## Обязательные файлы

В каждом активном адаптере должны быть:

1. `adapter.yaml`;
2. один основной entrypoint-файл инструкций;
3. локальные справочные файлы только при реальной необходимости;
4. install, usage или platform-specific config только там, где они реально нужны способу доставки.

## Манифест адаптера

`adapter.yaml` — это лёгкий inventory-файл адаптера.

В нём должно быть указано:

- id адаптера и тип deliverable;
- статус: `planned`, `active` или `deprecated`;
- какие файлы `core/` являются источником правды;
- какой файл инструкций считается главным;
- какие локальные файлы являются производными от `core/`;
- какие supporting docs и platform-specific config используются;
- что должно быть проверено перед выпуском.

## Что обязательно остаётся в `core/`

Каноническими в `core/` остаются:

- назначение и область применения;
- целевой тон и антицели;
- рабочий процесс;
- базовые правила и жёсткие ограничения;
- жанровые ориентиры;
- правило выдачи;
- таксономия паттернов;
- логика и философия примеров;
- кейсы и рубрика оценки.

Адаптеры могут переупаковывать этот материал, но не могут становиться его новым источником правды.

## Что адаптеры могут менять

Адаптеры могут менять только упаковку:

- файловую структуру;
- front matter и metadata;
- формулировки активации или ручного вызова;
- install и usage-инструкции;
- platform-specific config;
- локальные правила обращения к `references/`;
- сокращённую или перегруппированную подачу справочного материала.

Важно: это послабление относится к упаковке и справочным материалам. Основной entrypoint нельзя упрощать, сокращать или перестраивать так, чтобы менялся workflow, жёсткость ограничений, правило выдачи или условия обращения к reference-файлам.

## Что адаптеры не могут менять

Адаптеры не могут:

- менять редакторский стандарт;
- смягчать жёсткие ограничения;
- менять целевой тон;
- перестраивать workflow;
- вводить новое допустимое поведение, конфликтующее с `core/`;
- придумывать платформенные цели переписывания, которых нет в `core/`;
- заменять общий слой оценки своим локальным.

Если изменение влияет на редакторское поведение, сначала обновляйте `core/`.

## Держите адаптеры тонкими

Здоровый адаптер обычно выглядит так:

- один основной entrypoint;
- от нуля до двух локальных справочных файлов;
- немного install, usage или config-документации.

Если адаптер начинает жить своими правилами, примерами или локальной оценкой отдельно от `core/`, репозиторий начинает дрейфовать.

## Правило выпуска

Не держите адаптер в статусе `active`, если он не синхронизирован с:

- `core/prompt-spec.md`;
- `core/patterns.md`;
- `core/examples.md`;
- `core/eval-cases.md`;
- `core/eval-rubric.md`.

Оценка по умолчанию остаётся общей для обоих адаптеров. Не нужно копировать её в адаптеры без необходимости.
