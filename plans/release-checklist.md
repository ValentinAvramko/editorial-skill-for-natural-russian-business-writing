# Release-checklist перед выпуском

Это короткий внутренний чеклист перед любым следующим релизом.

Он не заменяет `adapters/smoke-checklist.md`, а собирает в одном месте финальные действия по содержанию, проверке и публикации.

## 1. Зафиксировать содержимое релиза

- [ ] Проверить, что редакторская логика менялась только через `core/`, а не напрямую в `adapters/`.
- [ ] Запустить `python scripts/build_adapters.py check` и убедиться, что `SKILL.md`, `PROMPT.md` и локальные `references/` не ушли в drift.
- [ ] Просмотреть `README.md`, `adapters/README.md`, install / usage-документацию и убедиться, что они соответствуют текущей структуре репозитория.

## 2. Подтвердить качество перед выпуском

- [ ] Пройти [`adapters/smoke-checklist.md`](../adapters/smoke-checklist.md) для `skill` и `prompt`.
- [ ] Если в релиз входят заметные изменения поведения, дополнительно прогнать `core/eval-cases.md` по `core/eval-rubric.md`.
- [ ] Убедиться, что оба deliverable-слоя сохраняют смысл, не придумывают детали и не расходятся по редакторской политике.

## 3. Подготовить публикацию

- [ ] Определить версию релиза и зафиксировать её в одном формате, например `v1.0.1`, `v1.1.0` или `v2.0.0`.
- [ ] Сформулировать короткие release notes для этой версии: что это за выпуск, что в него входит и для кого предназначены `skill` и `prompt`.
- [ ] Проверить, что версия релиза везде называется одинаково.
- [ ] Убедиться, что рабочее дерево чистое или содержит только осознанные файлы для релиза.

## 4. Выпустить релиз

- [ ] Сделать финальный commit с релизными правками.
- [ ] Запушить актуальную ветку в `origin`.
- [ ] Создать тег выбранной версии и запушить его.
- [ ] Открыть GitHub Release и вставить подготовленные release notes.

## 5. Проверить результат после публикации

- [ ] Убедиться, что на GitHub видны и commit, и тег нужной версии, и текст релиза.
- [ ] Открыть README и ключевые ссылки из release notes глазами пользователя.
- [ ] После публикации обновить рабочий план и перейти к следующему пункту.
