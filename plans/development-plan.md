# План развития

## Что уже сделано

- `core/` оформлен как единый источник правды для редакторской логики.
- Базовая редакторская спецификация, паттерны, примеры, eval-кейсы и рубрика разнесены по отдельным файлам.
- `adapters/skill/SKILL.md` и prompt-артефакты синхронизированы с `core/prompt-spec.md`.
- Для `adapters/prompt/` введены два entrypoint-а: самодостаточный quick-chat prompt и отдельный тонкий instructions-слой для проектов / Gems / knowledge-based сценариев.
- Prompt-слой переведён на плоскую структуру с уникальными именами reference-файлов рядом с entrypoint-ами.
- Для skill-а сохранён единый переносимый формат для Codex и Claude Code.
- Добавлен короткий smoke-checklist для ручной проверки `skill` и `prompt` перед выпуском.
- Добавлен лёгкий eval-runner для повторяемых ручных прогонов по `core/eval-cases.md`.

## Что было решено по архитектуре

- `core/` остаётся единственным местом, где меняется редакторская политика.
- `adapters/` содержит только готовые deliverable-пакеты без служебной документации и внутренних манифестов.
- Служебный слой сборки вынесен в `build/`.
- Всё, что относится к тестированию и проверке, вынесено в `test/`.
- Install-инструкции и пользовательские сценарии использования сведены в `README.md`, чтобы не плодить дублирующиеся документы.
- Prompt-адаптер разделён на два сценария доставки: один самодостаточный файл для обычного чата и один instructions-файл для settings-слоя, который требует обязательного использования основного prompt-файла и отдельно загруженных reference-файлов.
- Для prompt-адаптера выбрана плоская файловая структура с уникальными именами, чтобы файлы было удобнее загружать в ChatGPT Projects и Gemini Gems.
- Решение по drift принято в пользу лёгкого вспомогательного build/check-скрипта: `build/build_adapters.py` пересобирает `SKILL.md`, prompt entrypoint-ы и локальные reference-файлы из `core/`.

## Ближайшие шаги

- При новых редакторских изменениях сначала обновлять `core/`, потом прогонять `python build/build_adapters.py sync`.
- Поддерживать smoke и eval в `test/` как единое место для быстрой проверки quick-chat и file-backed сценариев перед выпуском.
